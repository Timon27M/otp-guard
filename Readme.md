# Описание модулей:

## Модуль аутентификации

Данный модуль отвечает за полный цикл управления учетными записями и сессиями пользователей: от регистрации до безопасного выхода из системы. Он построен на основе JWT-токенов (RS256) с механизмом их обновления.

### Как пользоваться сервисом

1. Зарегистрируйте нового пользователя через эндпоинт `/auth/register`.
2. Выполните вход через `/auth/login`. В ответе сервер установит две `HttpOnly` cookie:
    - `access_token` — время жизни 15 минут, используется для доступа к защищённым ресурсам.
    - `refresh_token` — время жизни 7 дней, используется для получения новой пары токенов.
3. При обращении к защищённым эндпоинтам браузер автоматически отправляет cookies. Если `access_token` истёк, фильтр безопасности обновит пару токенов прозрачно для пользователя.
4. Для завершения сессии используйте эндпоинт `/auth/logout` — cookies будут очищены, а refresh-токен аннулирован на сервере.

### API-эндпоинты и сценарии использования

| Метод | Эндпоинт | Назначение |
|-------|----------|------------|
| `POST` | `/auth/register` | Регистрация нового пользователя |
| `POST` | `/auth/login` | Вход в систему, установка токенов в cookies |
| `GET` | `/auth/logout` | Выход из системы, очистка cookies и аннулирование токенов |

**POST /auth/register**

Регистрирует нового пользователя. Проверяет уникальность логина, email и телефона.

Тело запроса:
```json
{
  "login": "testuser",
  "email": "test@example.com",
  "phone": "+79001234567",
  "password": "password123"
}

POST /auth/login

Авторизует пользователя. Требует отсутствия действующих токенов в cookies.

Тело запроса:
json

{
  "login": "testuser",
  "password": "password123"
}

GET /auth/logout

Завершает сессию пользователя. Аннулирует refresh-токен в базе и очищает cookies.
```

## Модуль управления пользователями

Данный модуль предоставляет функциональность для управления учётными записями: обновление собственной роли, получение списка всех пользователей и удаление пользователей по логину. Часть операций доступна только администраторам.

### Как пользоваться сервисом

1. Авторизованный пользователь может изменить свою роль через `/users/update-role`. После смены роли потребуется повторный вход.
2. Администратор может получить список всех пользователей через `/users`.
3. Администратор может удалить любого пользователя по логину через `/users/delete`. Удалить самого себя нельзя.

### API-эндпоинты и сценарии использования

| Метод | Эндпоинт | Доступ | Назначение |
|-------|----------|--------|------------|
| `PATCH` | `/users/update-role` | Авторизованные | Обновление роли текущего пользователя |
| `GET` | `/users` | `ADMIN` | Получение списка всех пользователей |
| `DELETE` | `/users/delete` | `ADMIN` | Удаление пользователя по логину |

**PATCH /users/update-role**

Позволяет текущему пользователю изменить свою роль. После успешного обновления роли принудительно выполняется выход из системы.

Параметры запроса:

| Параметр | Тип | Обязательный | Описание |
|----------|-----|--------------|----------|
| `role` | String | Да | Новая роль (`USER`, `ADMIN`) |

**GET /users**

Возвращает список всех зарегистрированных пользователей.

**DELETE /users/delete**

Удаляет учётную запись по логину. Нельзя удалить самого себя.

Параметры запроса:

| Параметр | Тип | Обязательный | Описание |
|----------|-----|--------------|----------|
| `login` | String | Да | Логин удаляемого пользователя |

## Модуль конфигурации OTP

Данный модуль позволяет администраторам просматривать и изменять глобальные настройки одноразовых кодов (OTP): длину кода и время его жизни. Все эндпоинты доступны только пользователям с ролью `ADMIN`.

### Как пользоваться сервисом

1. Администратор может получить текущие настройки OTP через `GET /otp-config`.
2. Администратор может изменить длину генерируемого кода через `PATCH /otp-config/update/code-length`. Допустимые значения: от 4 до 10.
3. Администратор может изменить время жизни кода (в минутах) через `PATCH /otp-config/update/lifetime-minutes`. Допустимые значения: от 2 до 10.

### API-эндпоинты и сценарии использования

| Метод | Эндпоинт | Доступ | Назначение |
|-------|----------|--------|------------|
| `GET` | `/otp-config` | `ADMIN` | Получение текущих настроек OTP |
| `PATCH` | `/otp-config/update/code-length` | `ADMIN` | Изменение длины OTP-кода |
| `PATCH` | `/otp-config/update/lifetime-minutes` | `ADMIN` | Изменение времени жизни OTP-кода |

**GET /otp-config**

Возвращает текущую конфигурацию OTP: длину кода, время жизни в минутах, дату последнего обновления и ID пользователя, внёсшего изменения.

**PATCH /otp-config/update/code-length**

Изменяет длину генерируемого OTP-кода.

Параметры запроса:

| Параметр | Тип | Обязательный | Ограничения |
|----------|-----|--------------|-------------|
| `codeLength` | Integer | Да | От 4 до 10 включительно |

**PATCH /otp-config/update/lifetime-minutes**

Изменяет время жизни OTP-кода в минутах.

Параметры запроса:

| Параметр | Тип | Обязательный | Ограничения |
|----------|-----|--------------|-------------|
| `lifetimeMinutes` | Integer | Да | От 2 до 10 включительно |

## Модуль OTP

Данный модуль отвечает за генерацию и проверку одноразовых кодов (OTP) с отправкой через различные каналы связи: email, SMS или Telegram. Коды привязаны к операциям, имеют ограниченное время жизни и одноразовое использование.

### Как пользоваться сервисом

1. Авторизованный пользователь запрашивает генерацию OTP-кода через `/otp/generate`, указывая название операции и канал доставки.
2. Сервис создаёт операцию, генерирует код, хеширует его и сохраняет в базе. Оригинальный код отправляется пользователю через выбранный канал.
3. Пользователь вводит полученный код и отправляет его на проверку через `/otp/check`, передавая ID операции и сам код.
4. Сервис проверяет принадлежность кода пользователю, его статус (не использован, не истёк) и правильность. При успехе код помечается как использованный.
5. Каждые 2 минуты планировщик автоматически помечает истёкшие коды как `EXPIRED`.

### API-эндпоинты и сценарии использования

| Метод | Эндпоинт | Назначение |
|-------|----------|------------|
| `POST` | `/otp/generate` | Генерация OTP-кода и отправка пользователю |
| `POST` | `/otp/check` | Проверка OTP-кода |

**POST /otp/generate**

Генерирует OTP-код для указанной операции и отправляет его через выбранный канал. Возвращает ID операции, который нужен для проверки кода.

Тело запроса:

```json
{
  "operationName": "CONFIRM_EMAIL",
  "channel": "EMAIL" или "SMS" или "TELEGRAM"
}
```

**POST /otp/check**

Проверяет OTP-код для указанной операции. При успешной проверке код становится недействительным для повторного использования.

Тело запроса:
json

{
  "operationId": "uuid-операции",
  "otpCode": "123456"
}

### 1) Для SMS нужно запустить симулятор (SMPP Simulator), передать в файле resources/sms.properties в smpp.port=2775 порт.

### 2) Для email нужно также заполнить файл resources/email.properties

### 3) Для telegram нужно создать бота после чего заполнить файл resources/telegram.properties
